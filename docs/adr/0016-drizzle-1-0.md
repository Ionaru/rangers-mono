# Drizzle moves to 1.0 (release candidate), while the database is still empty

`drizzle-orm` and `drizzle-kit` are pinned to **1.0.0-rc.4**, the `rc` dist-tag. This reverses the central pin of [ADR 0008](0008-drizzle-orm.md), which said to stay on 0.45.x until 1.0 landed, and it corrects two claims that ADR made about how the upgrade works. Everything else in 0008 stands: Drizzle is still the toolkit, postgres.js is still the driver, migrations are still generated by drizzle-kit and applied by the runtime migrator as a one-shot.

As a knock-on, Better Auth must be **1.7.0-rc.1** or later: stable `better-auth@1.6.23` pins its Drizzle adapter's peer at `drizzle-orm: ^0.45.2`, and only `@better-auth/drizzle-adapter@1.7.0-rc.1` widens it to `^0.45.2 || >=1.0.0-rc.1 <2.0.0` (better-auth#6766).

## Why now, rather than when 1.0 ships

Because the cost of this upgrade is a function of what is in the database, and right now the answer is nothing.

1.0's breaking change that actually touches us is the migrations-folder restructure, and the matching upgrade of `drizzle.__drizzle_migrations` from `(id, hash, created_at)` to `(id, hash, created_at, name, applied_at)`. That upgrade runs against production, once, whenever we do it. Today the five deployed tables hold zero rows: nothing in the repo contains a single `insert()`, so there is nothing to lose and every claim below could be checked against a database in exactly production's state. The next phase is Identity, which imports roughly 100 TeamSpeak links, and ADR 0011 calls those links the only irreplaceable data in the system, the one thing worth a week of the unit's time to recreate. Doing the restructure after that means doing it on top of the data we cannot afford to lose, to buy nothing.

So the choice is not "RC now or stable later". It is "restructure an empty database now, or the only valuable one later". 1.0 has been in RC since May and rc.4 has been out a fortnight; the risk of the RC is bounded and the risk of waiting is not.

There is a second, smaller reason: 1.0 retires a hazard 0008 documented at length. See below.

## What was verified, and how

0008 said its decision stood "on evidence, not on hope". Same standard. Every one of these was run against a local Postgres 17 that was already in exactly production's migration state: the five tables from `0000`, and a single `__drizzle_migrations` row with `hash = c597a6ae...` and `created_at = 1783805888324`, both written by the 0.45 migrator.

- **`drizzle-kit up` converts the folder, and only the folder.** `drizzle/0000_public_roulette.sql` + `meta/` became `drizzle/20260711213808_public_roulette/{migration.sql,snapshot.json}`. The folder's timestamp is derived from the journal's `when`, not from the clock. `meta/` is deleted. The snapshot goes from version 7 to 8, and its dialect string from `postgresql` to `postgres`.
- **The SQL is copied byte for byte.** sha256 before and after: `c597a6ae2b244b1df0614a58b7cdec19b907fe4fcd4888d6edc006a5ad44eb46`, unchanged. This matters more than it looks: that digest is what the production row is keyed on.
- **The conversion is faithful.** `drizzle-kit generate` against an unchanged `schema.ts` reports "No schema changes, nothing to migrate" and emits nothing. This was the one property that could not be settled by reading the source, because it is a semantic-equivalence question between two code paths (the v7-to-v8 snapshot converter, and the serializer that reads `schema.ts`). Had it emitted anything, that SQL would have run `ALTER`/`CREATE` against tables that already exist in production. It emits nothing, and CI's existing migration-drift gate now enforces that for free.
- **The runtime migrator upgrades the tracking table without re-applying anything.** It matched the existing row, added `name` and `applied_at`, set `name = '20260711213808_public_roulette'` and left `applied_at` NULL (deliberately: a NULL marks a row backfilled during the upgrade, as distinct from one applied by this migrator). The table still holds exactly one row, and all 40 columns across the five tables are byte-identical to before.
- **The new codec system decodes our columns unchanged.** 1.0 rewrote Postgres value decoding wholesale. Every timestamp we have is `timestamptz` and `operation.date` is a `date`, so both were round-tripped: a summer instant (CEST, the offset a naive `timestamp` would drop) comes back as the same instant, and `date` comes back as the same string with no timezone shift.
- **drizzle-kit 1.0-rc.4 runs on Deno 2.9.2.** `up` and `generate` both work. drizzle-team/drizzle-orm#5079 is open against Deno, but it is about `push`, which this project does not use, and the two independent requirements 0008 named (`nodeModulesDir: "auto"` at the root, and `drizzle-kit` declared in `packages/db/deno.json` imports) are still both required.

## The bug this found

`packages/db/client.ts` called `drizzle(sql)`, passing the postgres.js client positionally. 1.0 removed that overload, and it does not fail cleanly. The new `drizzle()` destructures its first argument looking for `client` or `connection`, finds neither on a postgres.js `Sql` (which is a function, not an options object), and falls through to opening a **new** connection from the `PGHOST` / `PGDATABASE` environment defaults. Our client would be silently discarded, and with it the `max: 1` and the lock and statement timeouts that `migrate.ts` sets and that ADR 0013 leans on: a migrator quietly pointed at a different database.

The types reject it, so `deno task check` catches it. It is called out here because the correct form, `drizzle({ client: sql })`, looks like needless ceremony and is exactly the kind of thing a later reader "simplifies" back.

## What this corrects in ADR 0008

Two of its claims are wrong, and both are the kind that get acted on:

- **"`drizzle-kit up` ... mutates production state: it adds `name` and `applied_at` columns to `drizzle.__drizzle_migrations` and rematches the existing rows. Take a dump first."** `drizzle-kit up` does not touch the database at all. It reads and writes files. The tracking-table upgrade is performed by the **ORM's** `migrate()`, in its own transaction, immediately before it applies anything. (Take the dump anyway. ADR 0013 already does, on every deploy.)
- **"Pending-ness is a timestamp high-water mark, not a hash set."** Not any more. 1.0's migrator reads every applied row and filters the local migrations by **set membership on the migration name**. This retires, entirely, the first of the two silent hazards 0008 warned about: a migration whose timestamp predates an already-applied one is no longer skipped forever, so a branch that sat too long, a revert-and-re-land, or a cherry-pick is no longer a way to lose a migration with no error and exit 0. The second hazard survives unchanged: **editing an already-shipped `.sql` is still a no-op**, because the name is already in the set. Migrations are still immutable once merged. Fix forward.

The rest of 0008's "What the migrator actually guarantees" still holds: all pending migrations still run in **one transaction** (so a failed migration still rolls back completely, and `CREATE INDEX CONCURRENTLY` is still unavailable, and we still do not need it); the SQL that runs is still the SQL baked into the image; and there is still **no down migration**, so the only undo remains the pre-migration dump.

## The guardrails paid off

0008 adopted two rules specifically to shrink this upgrade, and they did their job. Neither of 1.0's other two breaking changes applies to us:

- **No `relations()` and no `.query`.** 1.0 removed the v1 relational query builder from postgres outright. We never adopted it, so there is nothing to port. `client.ts` passes no `{ schema }` to `drizzle()`, which is what made the forbidden surface unreachable in the first place, and that same line is now what keeps Better Auth's adapter on its plain-query path.
- **No global `casing` option.** 1.0 reworked it from a `drizzle({ casing })` option into `snakeCase.table` / `camelCase.table` imports. We never used it.

What remained was the migrations-folder restructure, exactly as 0008 predicted, and it is a `drizzle-kit up` and a commit.

## Considered and rejected

- **Wait for 1.0.0 final.** The obvious answer, and the one 0008 gave. Rejected on timing, not on principle: the thing that makes this upgrade cheap is an empty database, and the very next phase fills it. Waiting does not reduce the work, it just moves it on top of the data that matters.
- **Regenerate a single fresh `0000` instead of converting.** Tempting, since the database is empty and a clean history is nicer than a converted one. Rejected because it breaks the tracking-table upgrade: a regenerated migration gets a new folder timestamp and possibly different SQL bytes, so neither of the migrator's two matching strategies finds the existing production row, and the upgrade throws rather than guessing. Converting preserves both, which is precisely what `up` is for. (It stays the fallback if the conversion had failed its gate, at the cost of a documented one-time drop of the empty tables.)
- **Stay on 0.45.x and drop Better Auth's Drizzle adapter**, wiring Better Auth to its own Kysely-backed Postgres adapter instead. Would decouple the two upgrades, and adds a second query builder and a second migration system to a project whose whole survival argument is a small tooling surface (ADR 0006). Rejected.

## Consequences

- **The migrations folder is now one directory per migration** (`<UTC timestamp>_<name>/migration.sql` + `snapshot.json`), and `meta/_journal.json` no longer exists. The worker image still carries `packages/db/drizzle/` for the same reason as before: the runtime migrator reads it from disk.
- **Both halves of the toolchain refuse to run against the old format** rather than mis-handling it. `drizzle-kit generate` says the folder is outdated and to run `up`; the migrator throws on finding `meta/_journal.json`. A forgotten `up` is a loud failure, not a silent one.
- **`deno task --cwd packages/db up` exists** and should never need running again. It is kept because it is the way back if the folder is ever restored from a pre-1.0 checkout.
- **Better Auth is pinned to an RC too**, for the peer range. Two RC pins now, not one.
- **These are release candidates, in production, for a volunteer-run unit.** Accepted knowingly, and bounded by three things: the database is empty, the deploy dumps before every migration (ADR 0013), and the whole path above was verified end to end against production's exact migration state before it shipped.
- **Revisit when 1.0.0 final ships**: it should be a version bump in `packages/db/{package.json,deno.json}` and `apps/web/deno.prod.json`, and nothing else. The same goes for Better Auth 1.7.0.
