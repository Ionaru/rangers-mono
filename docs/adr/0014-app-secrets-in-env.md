# Application secrets live in `.env`; the database keeps its file-based secrets

Every secret except two reaches `web` and `worker` as plain environment, from a git-ignored `.env` on the box that Compose loads with `env_file:`. The two exceptions are `postgres_password` and `database_url`, which stay Docker Compose file-based `secrets:` in `./secrets/`.

Correspondingly, a mounted `X_FILE` now **beats** a directly-set `X` in `packages/config`. That order used to run the other way.

## Why

The convention this replaces (every secret is a file-based Compose secret) is the more defensible one on a threat-model slide, and it is what IMPLEMENTATION §2 said from the start. It costs four edits per secret: a file in `./secrets/`, an `.example` beside it, an entry in the top-level `secrets:` map, and an `X_FILE` line in each service that reads it. That is the price of adding one webhook URL. At Phase 1 there were two secrets and the price was invisible. The remaining phases add six more (Discord client secret, bot token, session key, worker token, TeamSpeak user and password) and it stops being invisible: rotating a compromised bot token would mean touching three files under time pressure, and the failure mode of a convention that tedious is that someone quietly does not follow it.

The thing being bought is narrower than it looks. File-based secrets keep a value out of `docker inspect` and out of `/proc/1/environ`. They do **not** keep it off the box, out of the repo, or out of an image: `.gitignore` and `.dockerignore` do that, and they do it for both mechanisms equally. So what is actually traded away is resistance to an attacker who can already run `docker inspect` on the host, which is an attacker who can also `cat` the `./secrets/` directory the other mechanism stores its files in. On a single-tenant box that we do not even own (ADR 0005), that is a small delta, and a smaller one than the risk of a rotation nobody performs.

The database is exempt because it is already deployed and working. Moving a live credential to prove a point is how a green deploy becomes a red one, and the argument above is about the cost of *adding* secrets, not about the two that are already paid for. `postgres_password` has a second, harder reason: the postgres image refuses to start when both `POSTGRES_PASSWORD` and `POSTGRES_PASSWORD_FILE` are set, so its service must never see an `env_file` that might carry the plain key.

The precedence inversion is not a separate decision, it is what makes this one safe. `.env` holds `DATABASE_URL=postgres://7r:7r@localhost:5432/7r`, because the host-side tasks (`deno task migrate`, `web:dev`) need it. The moment Compose loads that same file into `web` and `worker`, it collides with the `DATABASE_URL_FILE` those services get from the mounted secret, and under the old order the plain value won: both containers would have dialled `localhost:5432` inside their own network namespace, found nothing, and crash-looped. The deploy would have discovered this at the `migrate` step, which runs *after* `docker compose stop web worker` (ADR 0013) has already taken the site down. Making the mounted file win removes the whole class by construction, and it costs nothing, because no environment anywhere sets both `X` and `X_FILE` with the intent that `X` should win.

## Considered and rejected

- **Keep every secret file-based.** The status quo. Rejected on the cost of adding the eighth secret, not the second: four edits per value, borne by one person in spare time, to defend against an attacker who already has the box.
- **Throw when both `X` and `X_FILE` are set**, the way the postgres image does. Sound in a vacuum, and it would have caught the `DATABASE_URL` collision loudly instead of silently. But that collision is not a mistake here, it is the normal state of every developer's `.env`, so this turns "you have a working local setup" into "your container refuses to boot" for everyone at once.
- **Strip `DATABASE_URL` from `.env` and let the host tasks read `./secrets/database_url`.** Removes the collision at its source, and breaks `deno task migrate` for anyone who has not populated `secrets/` yet, which is a worse first-run experience than the one being fixed.
- **`env_file:` with `required: false`.** Degrades gracefully when `.env` is absent, and needs Compose ≥ 2.24. The box is Windows + Docker Desktop and its Compose version is not pinned by anything we control (contrast the Deno version, which is pinned in four places precisely because this kind of drift bites). An older Compose rejects the `required:` key outright, and the deploy would discover that at the first compose command. A missing `.env` is now a missing secret store, which is worth failing on anyway, so the short syntax gives up nothing this project wanted, and it has no version floor at all. Revisit if the box's Compose is ever confirmed and pinned.
- **A real secret manager** (Vault, SOPS, Docker Swarm secrets). Infrastructure, and ADR 0005 puts infrastructure out of scope for a hundred-member Arma unit's handbook.

## Consequences

- **`.env` on the box is now load-bearing.** It is the secret store. It is untracked, so `git switch -d <sha>` in the deploy leaves it alone, and it is in `.dockerignore`, so it stays out of the build context. Nothing recreates it: if it is lost, it is lost, and the values must be gathered again from Discord, TeamSpeak and the secret manager between someone's ears.
- **Secrets are visible to anyone who can reach the Docker socket**, through `docker inspect`, `docker compose config`, and `/proc/1/environ` (including on stopped containers). That is the deal, made knowingly. Accordingly: `cd.yaml` pipes `docker compose logs` into a public Actions log, so a printer that ever echoes config would leak it. Do not add one.
- **A `$` in a secret is silently truncated** unless the value is single-quoted: Compose expands it in `env_file` values, so `SESSION_SECRET=abc$def` arrives as `abc`. A truncated `SESSION_SECRET` can still satisfy `.min(32)` while being far weaker than the one that was configured. `.env.example` says so at the top; there is no way to make Compose warn about it.
- **A blank key is not an unset key.** `env_file:` turns `FOO=` into `""`, which is defined, and would beat a schema default. `loadConfig` now reads `""` as absent, so `SYNC_DRY_RUN=` still defaults to `true` (safe) rather than crashing the worker.
- **The bot token in this file is `7R_Bot`'s, not the 2019 bot's.** The platform runs as its own Discord application (ADR 0015): the legacy bot's commands are ported, its account is not. The 2019 token never enters this file, or this repo. It does live in the old repo's git history, readable by everyone with access to it, which is one more reason not to inherit it. The defence for the token that *is* here is unchanged and is exactly one thing: `.env` is git-ignored. What is new is that rotation actually works now, because the credential is ours: one line and a restart, which is the cost this ADR was written to bring down.
- **The `migrate` service gets no `env_file`**, and neither does `postgres`. The migrator reads `DATABASE_URL` and nothing else, which the mounted secret already supplies.
