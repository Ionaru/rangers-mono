# Drizzle is the database toolkit (schema, queries, migrations)

**Status: the version pin is superseded by [ADR 0016](0016-drizzle-1-0.md), which moves to 1.0. The choice of Drizzle, of postgres.js, and of a one-shot runtime migrator all stand.** Two of the operational claims below are also wrong, and 0016 corrects them: `drizzle-kit up` does *not* touch the database, and pending-ness is no longer a timestamp high-water mark. Both are marked inline.

The `packages/db` package uses **Drizzle ORM** with **drizzle-kit** for migrations, over the **postgres.js** driver, on Deno via `npm:` specifiers.

## Why

Chosen by the owner. Drizzle fits the stack well: the schema is defined once in TypeScript and doubles as the migration source (`drizzle-kit generate`), so there is no separate schema language or codegen daemon; it runs on Deno 2 via `npm:drizzle-orm` + `npm:postgres`; it is light (no engine sidecar, unlike Prisma); and its inferred types flow into both `apps/web` and `apps/worker` through the shared package.

The full path was verified end to end: better-auth + `drizzleAdapter` + drizzle-orm + postgres.js on Deno against Postgres 17, including a complete Discord login. The decision stands on evidence, not on hope.

## Considered and rejected

- **Kysely**: a query builder only. Great typing, but no schema/migration source, so it would need a second tool for migrations. Drizzle covers both.
- **Prisma**: heavier (Rust engine + codegen) and historically awkward on Deno. More than this project needs.

## Consequences

- Schema lives in `packages/db/schema.ts`; queries live alongside it. Migrations are generated by drizzle-kit and applied by a **one-shot task**, never on app boot (a lesson from the legacy, which coupled entity discovery to startup).
- The Postgres driver is **postgres.js** (`drizzle-orm/postgres-js`), which works under Deno.
- ~~**Pin `drizzle-orm` to 0.45.x.**~~ **Superseded by ADR 0016**: the pin is now `1.0.0-rc.4`. The reasoning below (that 1.0's breaking changes hit the migration artifacts, which are the one thing that must keep working) is why 0016 verified every one of them against production's exact migration state before shipping.
- **Guardrail that shrinks the eventual 1.0 upgrade:** do NOT adopt the relational query builder (`.query` / `relations()`), and do NOT use the global `casing` option. The design uses only the core `pgTable` builder, so two of the three v1 breaking changes simply will not apply. That reduces the upgrade to the migrations-folder restructure. *(This worked: see ADR 0016. Both guardrails remain in force, and the relational builder is now not merely unused but removed from postgres in 1.0.)*
- **The 1.0 upgrade is not a repo-only change.** An earlier draft of this ADR said `drizzle-kit up` "largely automates" the restructure. It does not. At `drizzle-kit@0.31.10`, `up` only rewrites snapshots that are not yet at snapshot version 7, and ours already is, so it was a literal no-op here. The restructure is performed by `drizzle-kit@1.x`'s `up`, a different implementation. ~~and it **mutates production state**: it adds `name` and `applied_at` columns to `drizzle.__drizzle_migrations` and rematches the existing rows.~~ **Wrong, corrected by ADR 0016**: `drizzle-kit up` does not touch the database at all, it only rewrites files. The tracking-table upgrade is done by the *ORM's* `migrate()`. Take a dump first anyway (ADR 0013). The upside named here is real and did argue for upgrading promptly: v1's migrator applies all *missing* migrations by name rather than by timestamp high-water mark, which retires the silent-skip hazard below.
- **Two Deno-specific requirements, both independent, both required:** `"nodeModulesDir": "auto"` in `deno.json`, AND `drizzle-kit` declared in `deno.json` `imports`. Missing the second gives `Import "drizzle-kit" not a dependency`. Missing the first gives the actively misleading `Please install latest version of drizzle-orm`, even when it is installed and current.
- **Apply migrations with the runtime migrator** (`drizzle-orm/postgres-js/migrator`), never `drizzle-kit migrate`. `drizzle-kit migrate` drags tsx and three copies of esbuild (including the deprecated `@esbuild-kit/*`) into the Docker image.
- Pin the Deno version and commit `deno.lock`. The DB layer itself is not the fragile part: `drizzle-orm` and `postgres.js` each declare **zero** dependencies and have no Deno regression history. The place a Deno/npm regression would hurt most is `ts3-nodejs-library` -> `ssh2` -> `node:crypto` (see ADR 0006 and the implementation guide), which is where the interop risk actually lives.

## What the migrator actually guarantees

Four properties of `drizzle-orm/postgres-js/migrator` at 0.45.x. They are not documented upstream, they are load-bearing, and every sharp edge in the deploy is downstream of one of them. Read from the source (`pg-core/dialect.js`, `migrate()`).

- **All pending migrations run in one transaction.** So a failed migration rolls back *completely*: there is no half-applied schema, and the failure mode is always "new code against the old schema", never a torn one. The price is that no statement can escape that transaction, which rules out `CREATE INDEX CONCURRENTLY` and friends. We do not need them and are unlikely to: `attendance_session` grows by roughly 1,700 rows a year, so a plain `CREATE INDEX` on it is a matter of milliseconds.

- ~~**Pending-ness is a timestamp high-water mark, not a hash set.**~~ **Superseded by ADR 0016.** True of 0.45.x, and the reason 0016 argued for upgrading promptly. On 1.0 the migrator reads every applied row and filters the local migrations by **set membership on the migration name**. What that changes, and what it does not:
  - ~~**A migration whose `when` predates an already-applied one is skipped forever.**~~ **Retired.** This was the real hazard: no error, no warning, exit 0, "migrations applied" printed, and CI could not catch it because CI never touches a database. It took a branch that sat too long, a revert-and-re-land, or a cherry-pick. On 1.0 an out-of-order migration is simply applied, because it is matched by name and not by date.
  - **Editing an already-shipped `.sql` is still a no-op**, on 1.0 exactly as on 0.45: the name is already in the applied set, so the file is never read again. **Migrations are immutable once merged. Fix forward.**

- **The SQL that runs is the SQL in the image.** `apps/worker/Dockerfile` copies `packages/`, and `migrate.ts` resolves the folder relative to its own module, so the `migrate` service reads the `.sql` and the journal baked into `worker:<sha>`. The checkout on the box is never read for this; it only decides which `compose.yaml` is in effect.

- **There is no down migration.** `drizzle-kit generate` does not produce one, and rolling the images back leaves the new schema in place while restoring the old code, which is usually worse than the problem. The only undo is the pre-migration dump (ADR 0013).
