# Drizzle is the database toolkit (schema, queries, migrations)

The `packages/db` package uses **Drizzle ORM** with **drizzle-kit** for migrations, over the **postgres.js** driver, on Deno via `npm:` specifiers.

## Why

Chosen by the owner. Drizzle fits the stack well: the schema is defined once in TypeScript and doubles as the migration source (`drizzle-kit generate` / `migrate`), so there is no separate schema language or codegen daemon; it runs on Deno 2 via `npm:drizzle-orm` + `npm:postgres`; it is light (no engine sidecar, unlike Prisma); and its inferred types flow into both `apps/web` and `apps/worker` through the shared package.

## Considered and rejected

- **Kysely** — a query builder only. Great typing, but no schema/migration source, so it would need a second tool for migrations. Drizzle covers both.
- **Prisma** — heavier (Rust engine + codegen) and historically awkward on Deno. More than this project needs.

## Consequences

- Schema lives in `packages/db/schema.ts`; queries live alongside it. Migrations are generated by drizzle-kit and applied by a **one-shot task**, never on app boot (a lesson from the legacy, which coupled entity discovery to startup).
- The Postgres driver is **postgres.js** (`drizzle-orm/postgres-js`), which works under Deno.
- Pin the Deno version and integration-test the `npm:` interop early; the DB layer is the one place a Deno/npm regression would hurt most.
